# AI-Customer-Service-Agent

这是一个面向学习、课程展示和面试讲解的 AI 客服 Agent 项目。项目以传统工单系统为基础，接入 AI 服务，让客服处理流程具备自动分类、知识问答、回复建议、历史记忆和退款协助等能力。

本项目不是生产级系统，重点展示“业务系统 + Agent 工作流 + 知识库 + 记忆 + 前后端交互”的整体设计思路。

## 项目概览

系统由三部分组成：

- `workOrderFrontend-main/`：用户和管理员使用的 Web 页面。
- `workOrderBackend-main/`：业务后端，负责登录、工单、订单、退款、知识库文件和状态流转。
- `workOrderAI/`：AI 服务，负责分类、知识检索、回复建议、记忆和评测。

整体关系可以简单理解为：

```text
用户 / 管理员
  -> 前端页面
      -> Java 业务后端
          -> MySQL 业务数据
          -> Python AI 服务
              -> 知识库、历史案例、用户记录、模型能力
```

## 核心能力

- 用户提交反馈、查看反馈进度、查看已有订单和物流。
- 管理员处理工单，查看用户问题、历史回复和 AI 回复建议。
- AI 自动识别工单分类、优先级和用户情绪。
- 知识问答支持从本地资料中检索答案。
- 回复建议会结合当前工单、最近对话、知识库、用户记录和历史案例。
- 系统包含短期工单记忆、历史案例记忆和用户事实记忆，用于减少重复询问和重复建议。
- 电商退款场景支持用户从订单页发起退款申请，管理员审核退款方案。
- 提供本地评测能力，用于观察 Agent 改动前后的效果变化。

## 主要页面

用户侧：

- `我的反馈`：提交问题、查看处理进度、继续补充回复。
- `我的订单`：查看订单编号、商品、金额、物流状态，并发起退款申请。
- `知识问答`：面向用户的知识库问答入口。

管理员侧：

- `工单管理`：查看、筛选和处理用户反馈。
- `AI 回复建议`：根据工单上下文生成客服回复草稿。
- `退款审核`：审核退款申请，确认退款处理结果。
- `知识库管理`：维护本地知识文档。

## Agent 设计思路

本项目没有把大模型当作单次问答接口使用，而是把它放进工单处理流程中：

1. 先理解当前问题属于哪类业务场景。
2. 再根据场景选择知识库、用户记录、历史案例或退款工具。
3. 生成客服可参考的回复。
4. 对回复做基本自检，不合格时尝试改写或降级。
5. 工单结束后，将有价值的信息沉淀为记忆，供后续工单复用。

退款相关能力有明确职责边界：Agent 负责理解意图、补充追问和生成说明；订单归属、退款资格、金额计算、审核状态和执行结果以 Java 后端为准。

## 数据与记忆

系统中的记忆大致分为三类：

- 工作记忆：保存当前工单的摘要、已确认事实、已尝试步骤和待确认问题。
- 案例记忆：沉淀已解决工单，后续遇到相似问题时提供参考。
- 用户事实记忆：保存用户明确表达或稳定出现的设备、环境、偏好和高频问题。

知识库和案例记忆相互独立：知识库更像产品资料和常见问题，案例记忆更像客服处理经验。

## 演示场景

项目当前适合展示这些流程：

- 用户提交普通问题，管理员使用 AI 回复建议处理。
- 用户提问知识库问题，系统从资料中检索并回答。
- 用户查看订单，从订单页申请退款。
- Agent 根据退款信息追问订单号、判断退款类型，并交给后端规则校验。
- 管理员在工单详情中审核退款申请。
- 通过评测对比不同 Prompt、路由和记忆策略的效果。

workOrderAI/data/         本地知识资料和向量库数据

docker-compose.yml        本地全栈编排

## 本地运行

本项目支持本地分服务运行，也支持 Docker Compose 启动。具体命令和本机环境路径以 `AGENTS.md` 中的说明为准。

常见服务端口：

- 前端：`5173` 或 Vite 自动分配端口
- Java 后端：`8080`
- Python AI 服务：`8003`
- MySQL：本地或 Docker 中的 MySQL 实例
